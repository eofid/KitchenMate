# Отчет по Лабораторной работе №3: Исследование архитектуры проекта KitchenMate

## Часть 1. Проектирование архитектуры (To Be)
На высоком уровне абстракции система спроектирована как классическое многослойное приложение.

1. **Тип приложения:** Desktop GUI (на базе Tkinter).
2. **Стратегия развертывания:** Standalone (локальная установка).
3. **Технологии:** Python 3.13, SQLite (БД), TheMealDB (внешний API).
4. **Качество:** Высокая отзывчивость интерфейса и надежное хранение данных.
5. **Сквозная функциональность:** Единая система авторизации и централизованная обработка ошибок API.

## Часть 3. Сравнение и рефакторинг

### 1. Сравнение архитектур
Схема **"As Is"** наглядно демонстрирует выполнение принципов из схемы **"To Be"**:
* Интерфейс (UI Layer) полностью отделен от логики работы с БД.
* Все запросы проходят через центральный контроллер `KitchenMateApp`.

### 2. Отличия и причины
* **Отличие:** В текущем коде авторизация вынесена в JSON-файл, в то время как остальные данные хранятся в SQLite.
* **Причина:** Использование JSON позволило быстро реализовать MVP (минимально жизнеспособный продукт) во время первого спринта.

### 3. Пути улучшения (Рефакторинг)
1. **Унификация БД:** Перенос данных пользователей из JSON в основную базу SQLite.
2. **Абстракция API:** Создание отдельного класса-провайдера для работы с интернет-рецептами (уменьшение связанности кода).
3. **Интерфейсы:** Использование базового класса для всех экранов для исключения дублирования кода импортов стилей.

## Вывод
Архитектурное решение проекта KitchenMate является масштабируемым. Выбранная модель "Звезда" позволила безболезненно добавить экран логина и расширить систему поиска в ходе разработки.
